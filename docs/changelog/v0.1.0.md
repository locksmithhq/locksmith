# v0.1.0 — 2026-04-23

## Added

### Social Login (OAuth2 Social Providers)
- Support for Google, GitHub, Facebook, Discord, and LinkedIn as identity providers via [goth](https://github.com/markbates/goth).
- New admin tab in the client details screen to configure and enable/disable social providers per client.
- Social login button automatically appears on the login page when at least one provider is enabled for the client.
- Accounts created via social login are linked to the Locksmith account by provider user ID. If an account with the same e-mail already exists, the social provider is linked to it.

### PKCE (Proof Key for Code Exchange)
- New `require_pkce` flag on OAuth clients. When enabled, the authorization code flow rejects requests without a valid `code_challenge`.
- Only `S256` is accepted as `code_challenge_method`. `plain` is not supported.
- Configurable via the admin UI (client details → Security tab) and via the API.

### Secrets Encryption at Rest (AES-256-GCM)
- `oauth_clients.client_secret` and `oauth_client_social_providers.(client_key, client_secret)` are now encrypted in the database using AES-256-GCM.
- Encryption is transparent — values are decrypted on read and encrypted on write.
- Existing plaintext values continue to work until the next write (backwards-compatible prefix detection).
- `client_secret` is masked in API responses (`****<last-4-chars>`). The full value is returned only on creation.

### Secure Logout
- New `POST /logout` endpoint that revokes the refresh token and the associated session.

### Token Reuse Detection
- If a revoked refresh token is presented, the entire session chain is revoked immediately (RFC 6819 §5.2.2.3).
- A `[SECURITY] token_reuse_detected` log line is emitted with `account_id`, `client_id`, and `token_hash`.

### Timing Attack Protection
- `hasher.PerformDummyVerify` runs a full Argon2id verification on a pre-computed dummy hash when an account is not found, equalizing response time and preventing user enumeration (CWE-204).

### Production Error Mode
- Set `APP_ENV=production` to hide internal stack traces from API responses.
- In production, errors return only `message` and `error_id`. The full trace is written to the log for correlation.

### CI/CD & Infrastructure
- GitHub Actions workflows: CodeQL (Go + JavaScript) and Security SAST (`govulncheck v1.1.4`, `gosec v2.25.0`, `golangci-lint`, `npm audit`).
- Dependabot configured for Go modules, npm, and GitHub Actions (weekly).
- Multi-stage production `Dockerfile` (`scratch` base, non-root user, multi-arch `linux/amd64` + `linux/arm64`).
- `make build-prod` publishes to Docker Hub.
- `.dockerignore` added.

### Accounts Tab
- Server-side pagination and search on the Accounts tab (`v-data-table-server`).

---

## Changed

- **Seeder credentials** are no longer hardcoded. They are now provided via environment variables — see [Upgrade Guide](#upgrade-guide) below.
- **`pgweb`** service removed from `compose.yaml`.
- Password visibility toggle added to login, register, and auth views.
- PKCE validation enforced at both the `/authorize` (login) and `/token` endpoints.

## Security

| CVE / Issue | Description | Fix |
|---|---|---|
| CWE-312 | `client_secret` and social provider credentials stored in plaintext | AES-256-GCM at-rest encryption |
| CWE-204 | User enumeration via login response timing | Dummy Argon2id verify on missing account |
| RFC 6819 §5.2.2.3 | Refresh token reuse not detected | Session-wide revocation on reuse |
| — | Internal stack traces exposed in API errors | `APP_ENV=production` masking mode |

## Upgrade Guide

### 1. Run database migrations

Migrations run automatically on startup. All new columns have safe defaults — no data is lost.

### 2. Add required environment variables

```env
# Hides stack traces from API responses.
APP_ENV=production

# AES-256-GCM key — generate with: openssl rand -base64 32
ENCRYPTION_KEY=<your-32-byte-base64-key>

# Seeder (required on first boot or db reset)
SEED_ADMIN_EMAIL=admin@yourorg.com
SEED_ADMIN_PASSWORD=<strong-password>
SEED_APP_CLIENT_ID=<uuid>
SEED_APP_CLIENT_SECRET=<strong-secret>
SEED_APP_USER_PASSWORD=<strong-password>
```

> **`ENCRYPTION_KEY` is mandatory.** The application will refuse to process any encrypted field without it.

### 3. Re-encrypt existing secrets (one-time)

New writes are encrypted automatically. To re-encrypt all existing plaintext rows at once:

```bash
# From the api/ directory, with your .env loaded:
ENCRYPTION_KEY=<your-key> go run ./cmd/migrate-secrets/main.go
```

Expected output:
```
oauth_clients:                  N/N rows re-encrypted
oauth_client_social_providers:  N/N rows re-encrypted
Done.
```

The tool is idempotent — already-encrypted rows are skipped.

### 4. Update Docker Compose or `docker run` (if self-hosting)

For full deployment examples (Docker Compose and `docker run`) see [`docs/deployment.md`](../deployment.md).

**If upgrading an existing Compose setup**, add to the `api` service environment:

```yaml
- APP_ENV=${APP_ENV}
- ENCRYPTION_KEY=${ENCRYPTION_KEY}
- SEED_ADMIN_EMAIL=${SEED_ADMIN_EMAIL}
- SEED_ADMIN_PASSWORD=${SEED_ADMIN_PASSWORD}
- SEED_APP_CLIENT_ID=${SEED_APP_CLIENT_ID}
- SEED_APP_CLIENT_SECRET=${SEED_APP_CLIENT_SECRET}
- SEED_APP_USER_PASSWORD=${SEED_APP_USER_PASSWORD}
```

If you rely on `pgweb`, add it back in a local `compose.override.yaml`.

### 5. Review PKCE configuration

`plain` is now rejected. Check which clients have PKCE enabled:

```sql
SELECT id, name, client_id, require_pkce
FROM oauth_clients
WHERE require_pkce = true;
```

Update any application sending `code_challenge_method=plain` to use `S256`.
